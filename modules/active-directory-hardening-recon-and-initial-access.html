Active Directory Hardening - Recon & Initial Access

Introduction & Scenario

Hardening Active Directory (AD) and remediating findings from third-party penetration tests and internal assessments, along with applying patches for newly disclosed vulnerabilities, performing ongoing security maintenance, and hardening tasks related to Active Directory, frequently fall under the responsibility of systems administrators. These professionals are often simultaneously tasked with numerous operational duties. Active Directory is notoriously vulnerable, from legacy configurations and abuse of built-in functionality to misconfigurations caused during routine setup and configuration changes and significant vulnerabilities with working public exploits. There is a lot to manage and unpack, a lot that can go wrong, and introducing a change in one place can introduce a vulnerability in another. Dealing with Active Directory security in on-premises environments can feel like playing whack-a-mole or patching holes in a leaky boat only to have another leak spring up. Whatever the analogy, Active Directory security is not for the faint of heart. A firm grasp of the issues throughout the penetration testing cycle is imperative for maintaining a well-hardened environment that properly supports day-to-day operations.

Every penetration test or compromise starts with reconnaissance/enumeration. By cleaning up issues that allow for unauthenticated AD enumeration, we can slow an attacker down, but we must also deal with flaws that allow an unauthenticated attacker to gain initial access. The more difficult it is for an attacker to enumerate the environment or gain initial access, the noisier they will have to become, which increases their chances of being caught. However, we want to ensure that no initial access vectors exist. We will focus on the most common flaws seen across enterprises of all sizes, which are possible by an unauthenticated attacker who can gain internal network access.

This module, and the others to follow in this path, will simulate a systems administrator at Wyrmwood Mining Co., a mining and logistics company that has just undergone its first internal penetration test. Until now, the focus has been on perimeter security with an annual external penetration test and quarterly vulnerability scans. Due to recent changes in IT leadership, including bringing in a more technical CISO, there is a renewed focus on internal network security, namely related to Active Directory and how far an attacker or malicious insider could go in the network.

Pre-Windows 2000 Compatible Access Group

Legacy holdover configurations frequently persist in Active Directory environments, regardless of their size or maturity. A prominent example is the Pre-Windows 2000 Compatible Access group. Contrary to common assumptions, this is not a setting per se, but rather a built-in security principal intended to maintain backward compatibility with legacy Windows NT-based systems and applications.

The group was introduced when Windows 2000 was released to replace the Windows NT domain setup. This change moved from a flat directory to authenticate users and computers to the introduction of Active Directory with the concepts of domain trees and forests and the ability to create a hierarchical structure using Organizational Units (OUs) within a domain. This change allowed organizations to delegate permissions better and assign specific policies to a grouping of users or computers.

The release of Active Directory also unlocked the ability to set permissions at both the object level (users, groups, and computers) as well as attributes (description, department, job title, group membership, etc.) for each individual object. This granular access control model granted systems administrators the capability to read or write all attributes of any Active Directory object, granting much more fine-grained control over the domain. The Pre-Windows 2000 Compatible Access group is pre-populated with the Authenticated Users group with any new deployment of Active Directory, even up to Windows Server 2022 and 2025.

The group's description in Active Directory Users and Computers on a domain controller reads: "A backward compatibility group which allows READ access on all users and groups in the domain". This group's purpose is in the name; it was designed to be compatible with Windows NT, allowing an admin to grant permissions at the object level instead of using more granular ones. In other words, if a user is a member of this group, they are granted READ access to all user and group objects across the domain.

This was introduced to ensure older systems and applications could continue reading complete user and group lists without breaking during the transition to Active Directory in Windows 2000. However, what was practical 25 years ago now creates a security risk: any domain user can, by default, view most objects and their attributes (unless additional restrictions are implemented).

This aids in enumeration (i.e., making a user list for password spraying or finding a password in a user or computer description field). In other words, in 2000, Active Directory was created as a large, read-only database for non-admin users, and this functionality is still the standard in most organizations.

If we open ADSI Edit and the Advanced Security Settings for the domain object, we can see that the Pre-Windows 2000 Compatible Access group has Special permissions over users and groups in the domain. Read access from the top down, which is inherited down the entire hierarchy of OUs (applies to Descendant InetOrgPerson objects, Descendant Group objects, and Descendant User objects).

If we click Edit for Descendant User Objects, for example, we see that these Special permissions equate to read access over all properties.

Once again, this is why if, during a penetration test, the consultant is able to take over a standard domain user account (or obtain NT AUTHORITY\SYSTEM rights over a domain-joined computer), they can proceed with detailed enumeration of the entire domain to look for flaws and plan out attack paths.

You may be thinking, "Why is this important?" Well, when deploying a new AD forest using older versions of Windows Server (e.g., Windows Server 2003), you were explicitly asked if you would like to deploy to be compatible with pre-Windows 2000 versions.

When this was done, the Anonymous Logon identity and Everyone group were typically added as nested members of this group. This would grant both of these security principals read access over the entire Active Directory domain without needing to authenticate. Nowadays, it is less common to see the Anonymous Logon identity as part of this group, but it is still very common to see the Everyone group as part of it.

While this is a legacy holdover from the earlier days of AD, it is still present in many modern environments, even with higher domain functional levels, such as functional level 2012. This is because these types of legacy settings are "pulled up" when domain functional level upgrades are performed over time, but a full security cleanup is not performed. In domains created on domain controllers starting with Windows Server 2008, systems administrators were no longer prompted to decide whether or not to add the Everyone group to the pre-windows 2000 group. Keep in mind that this was never a purely default setting, even when running the Active Directory wizard on a Windows Server 2000 box. However, it was likely chosen more often than not to avoid potential issues. The help page on a Windows Server 2000 domain controller for Promoting a domain controller even mentioned that removing the Everyone group would be the more secure configuration, but this advice likely was not always heeded or not practical at the time.

If the Everyone group alone is found within the Pre-Windows 2000 Compatible Access group with no other settings in place within Group Policy, this is typically not enough to cause an issue. However, if the Anonymous Logon identity is a member of this group, then some major flaws will exist that an attacker can use to perform unauthenticated enumeration. There are various other conditions that are often present in Active Directory environments that we assess.

In the next section, we will examine three methods penetration testers use to perform unauthenticated recon if the environment supports them. While outlier configurations may exist where these issues are still present, we will cover the major whys and methods to fix each.

Unauthenticated Recon & Enumeration Weaknesses

Unauthenticated enumeration against Active Directory can lead to an attacker obtaining key data such as lists of all users and computers and attributes for both (such as the description field), which can sometimes contain passwords, groups, and the domain password policy. These data points cannot be typically be used to gain access (unless a valid password is uncovered in the description field), but combined, these can be used as input for a password spraying attack. With a list of valid users and the domain password policy, an attacker can craft passwords within the bounds of the password policy and perform password spray attempts at set intervals to avoid account lockout based on the account lockout threshold in the policy.

Remediating SMB Null Sessions

Luckily, the fix for this issue is not difficult. The first step is to remove the Everyone group from the Pre-Windows 2000 Compatible Access group. This typically solves the issue. However, a second check that should be made would be reviewing the Default Domain Controllers Policy in Group Policy and ensuring that Network access: Let Everyone permissions apply to anonymous users is disabled. This can be done by editing the policy and browsing to Computer Configuration --> Windows Settings --> Security Settings --> Local Policies --> Security Options, changing the setting to disabled, and then doing a gpupdate.

LDAP Anonymous Bind

An LDAP anonymous bind is another method that an attacker can use to obtain information from the Active Directory domain without requiring authentication. This can be used to pull data such as a listing of users, groups, computers, and user account attributes (i.e., the description field that sometimes contains passwords), along with the domain password policy. When an anonymous base bind is allowed by a domain controller, an attacker can obtain a significant amount of information about the domain without needing to know a base object. This data can then be used to mount a password spraying attack or ASREPRoasting attack, similar to when an SMB null session is allowed.

This was a default setting on Windows Server 2000 domain controllers, once again due to the Pre-Windows 2000 Compatible Access group. It was not the default beyond Server 2000 but can be pulled along over the years when upgrades in place are done. Below is a screenshot of the access control settings for all Users in a domain using ADSI Edit.

There are various other reasons an organization may have configured LDAP anonymous binds, such as legacy applications/services that needed to bind anonymously to look up users or groups, intranet portals that may have used anonymous binds to allow basic user lookup (i.e., a user directory), public or semi-public directory access, ease of development, etc. During a penetration test, the tester may find an anonymous bind that only allows for querying limited information, which may be due to an intentional configuration. Regardless of the reason, in modern AD environments, LDAP anonymous binds present risk and should be disallowed unless there is a clear business reason and other mitigating controls are in place.

Remediating LDAP Anonymous Bind

To remediate this fully, it is recommended that the dsHeuristics attribute value within the Directory Service object in AD be cleared. When this value is set to 0000002, anonymous binds are possible. Clearing this value or removing the 2 disables anonymous binds at the directory level.

This can be done manually by opening ADSI Edit (adsiedit.msc), right-clicking ADSI Edit --> Connect To -->, and choosing Configuration as the Naming Context.

Afterwards, browse to CN=Directory Service,CN=Windows NT,CN=Services,CN=Configuration,DC=yourdomain,DC=com, right-click on CN=Directory Service and select Properties.

Once here in the Attribute Editor, scroll down to find the dsHeuristics attribute and clear the value.

We should also ensure that ANONYMOUS LOGON does not have read access over the Users CN or other objects, including the domain object itself.

This can also be remediated with a single PowerShell script as follows.

# Obtain domain name, build LDAP path to the Directory Service object, connect as an ADSI object
$Dcname = Get-ADDomain | Select-Object -ExpandProperty DistinguishedName
$Adsi = 'LDAP://CN=Directory Service,CN=Windows NT,CN=Services,CN=Configuration,' + $Dcname
$AnonADSI = [ADSI]$Adsi

# Clear the dSHeuristics attribute 
$AnonADSI.Properties["dSHeuristics"].Clear()
$AnonADSI.SetInfo()

# Remove ANONYMOUS LOGON read access on CN=Users
$ADSI = [ADSI]('LDAP://CN=Users,' + $Dcname)
$Anon = New-Object System.Security.Principal.NTAccount("ANONYMOUS LOGON")
$SID = $Anon.Translate([System.Security.Principal.SecurityIdentifier])
$adRights = [System.DirectoryServices.ActiveDirectoryRights] "GenericRead"
$type = [System.Security.AccessControl.AccessControlType] "Allow"
$inheritanceType = [System.DirectoryServices.ActiveDirectorySecurityInheritance] "All"
$ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule $SID,$adRights,$type,$inheritanceType
$ADSI.PSBase.ObjectSecurity.RemoveAccessRule($ace) | Out-Null
$ADSI.PSBase.CommitChanges()

The script does the following:

  1. Obtains the domain DN, builds an LDAP path to the CN=Directory Service in the Configuration partition, and connects as an ADSI object for read/write access
  2. Clears the dSHeuristics attribute
  3. Removes explicit GenericRead permissions for ANONYMOUS LOGON on CN=Users to prevent anonymous LDAP binds for reading user objects from an unauthenticated standpoint

This attack works by default on Server 2000 and Server 2003 domain controllers if the Everyone group is added to the Pre-Windows 2000 Compatible Access group, which was often the case. In our tests, this is not the default behavior from Server 2008 domain controllers and onwards. Remediating RID Brute Forcing

This can be remediated by either setting Network access: Allow anonymous SID/name translation to disabled in the Default Domain Controllers policy in the Group Policy editor or by setting the registry key HKLM\SYSTEM\CurrentControlSet\Control\Lsa\RestrictAnonymous to 1 or 2. Ideally, this should be remediated via Group Policy to ensure the setting is applied to all domain controllers in the forest.

Recap

We've covered three legacy methods for enumerating data, such as a list of valid domain users and the password policy, among other data, that can be used as inputs for a password spraying attack. On their own, this is just enumeration data, but it is a first step for an attacker to gain a foothold if other issues exist, such as a weak password policy/legacy passwords that are set outside of the policy (i.e., accounts with passwords that never expire that had their password set years ago). Furthermore, if sensitive data is stored within user or computer attributes, such as passwords, this data could potentially be used to gain a foothold. Lastly, the user list could be used to perform an ASREPRoasting attack, which we will cover later in this module.

It is worth checking all domain controllers in the forest to see if any of these legacy settings exist and remediating them to tighten things up if there is no valid business reason for them to be in place.

Initial Access Weaknesses - Response Spoofing Attacks

Now that we've covered methods used by penetration testers for performing unauthenticated enumeration, we'll dive into common ways of obtaining initial access during an internal penetration test. These attacks leverage built-in network protocols/settings enabled by default unless hardened against.

LLMNR/NBT-NS Response Spoofing

LLMNR & NBT-NS (NetBIOS) spoofing/poisoning are well-known internal network attacks that still work to this day in many environments. LLMNR stands for Link-Local Multicast Name Resolution, while NBT-NS stands for the NetBIOS Name Service, NetBIOS being short for Network Basic Input/Output System. These legacy protocols are enabled by default and, like with many other issues, can remain in the network even after domain controller/domain functional level upgrades.

The LLMNR protocol allows hosts to perform name resolution with hosts on the same internal network segment without needing DNS. When a DNS name server request fails, Windows systems will attempt fallback name resolution using LLMNR and NBT-NS if these protocols have not been disabled. The primary issue with LLMNR and NBT-NS broadcasting within an internal network is that if a DNS name fails to resolve, the client (system) will send out an unauthenticated UDP broadcast to the entire local network segment, checking to see if any other system on the segment has the name that it's searching for. Since this broadcast process is performed unauthenticated to the entire network, any machine on the network can respond in place of the requested machine.

An attacker can position themselves on the network, listening for LLMNR/NBT-NS broadcasts using a tool such as Responder and spoof responses as the machine the client is attempting to authenticate to. During this authentication process, the client machine will send out an NTLMv2 password hash (sometimes NTLMv1 if this legacy protocol is enabled in the network), which is then captured on the attacker machine for offline password cracking or "relaying" attacks, which will be covered later in this section.

Remediating LLMNR/NBT-NS Response Spoofing

LLMNR & NBT-NS response spoofing attacks are relatively easy to remediate by disabling both protocols. If only LLMNR is disabled, then Windows will failover to NBT-NS for name resolution, so the attack will still be possible.

LLMNR

LLMNR can be disabled via Group Policy. The ideal way to do this is by creating a new GPO instead of disabling it via the Default Domain Policy. Creating a dedicated GPO such as Disable LLMNR will make it more easily auditable, easier to target specific OUs, easier to roll back, and avoids editing the domain-level baseline policy.

To disable this, we need to perform the following steps after creating a new GPO: Computer Configuration → Administrative Templates → Network → DNS Client → turn off multicast name resolution. First, right-click on Group Policy Objects and select New.

Next, give the GPO a name; we'll call it Disable LLMNR.

Next, right-click on the GPO and click edit. Browse to the setting Turn off multicast name resolution, click Enabled and then Apply.

Next, we need to link the GPO to specific OUs where we want it applied. Our lab has Workstations and Servers under the CORP OU.

First, we'll right-click on the CORP OU in Group Policy Management and select Create a GPO in this domain, and Link it here. In a non-lab environment, we would select an OU such as Workstations or Servers, etc. Right-click on Workstations and select Link an existing GPO.

Now that it's linked, we can wait for Group Policy to update, which by default is 90 minutes with an offset of 30 minutes, to ensure every computer in the domain isn't attempting to update simultaneously. Alternatively, we could force a Group Policy update from the command line using gpupdate /force or by right-clicking on the OU and selecting Group Policy Update.

With LLMNR and NBT-NS now disabled in the network, we are now protected against two of the most common network response spoofing attacks that can lead to a quick and easy foothold.

mDNS Response Spoofing

Another candidate for response spoofing attacks is Multicast DNS (mDNS), which is often overlooked. The classic penetration testing finding for successful response spoofing is typically along the lines of LLMNR/NBT-NS Response Spoofing. A client will receive the report, remediate both issues and the penetration tester will validate the fix. However, the following year, the tester is still able to perform a response spoofing attack to obtain password hashes or perform an NTLM relay attack. mDNS was introduced with Windows 10 1703. Apple developed the protocol as a means for performing local network name and service discovery without the need for a DNS server or user interaction. This is how Apple was able to make setup for AirPlay2- based services easy using the Bonjour service. mDNS is used by a myriad of devices and services by Microsoft, Google, Apple, Amazon, etc. It is used with smart TVs, printers, set-top boxes, wireless speakers, wireless screen mirroring, etc, via UDP port 5353. It is everywhere. Fire up Wireshark in your home network, and if you have any type of smart device, you will likely see it in action.

Remediating mDNS Response Spoofing

While there is a Registry value for disabling mDNS in the Windows DNS client resolver, it does not fully disable mDNS in a network because any operating system could have multiple mDNS resolvers running simultaneously. Since mDNS uses UDP port 5353 and not TCP, it is possible to have various services listening on that port simultaneously. If you run a command like the one below on your own system, you will likely see multiple different processes using UDP port 5353.

Get-NetUDPEndpoint -LocalPort 5353 | Select-Object LocalAddress,LocalPort,OwningProcess,@{ Name="ProcessName"; Expression={((Get-Process -Id $_.OwningProcess).Name )} }

It is possible to disable Windows' built-in mDNS resolver that is located within the Dnscache service (inside svchost.exe), which handles .local name resolution via Group Policy; this only puts a bandaid on the problem if other services are running such as Chrome, TeamViewer, Zoom, etc. These third-party apps, which are listening and broadcasting on UDP port 5353, implement their own mDNS stacks for discovering peers, sharing content, supporting cross-device syncing or casting, etc. So even if the Windows built-in mDNS resolver is disabled, stopping Windows from asking for mDNS, other applications in the network may still send mDNS queries or accept mDNS replies. If any of them perform .local lookups and accept spoofed replies, they are still vulnerable (i.e., Chrome).

Microsoft's official recommendation for locking down mDNS is to use the Windows Defender Firewall. This already existing tool can help organizations mitigate this issue in various ways. The options include:

  • Disable inbound mDNS in Windows Defender Firewall for ALL profiles (Public, Private, Domain). This will prevent all inbound mDNS traffic but could cause issues for remote workers
  • Disable mDNS just within the Domain profile, which will not affect remote workers but will block mDNS within the corporate network
  • Disable mDNS in Windows Defender Firewall for outbound traffic. This is only necessary where high security is required, as blocking inbound mDNS traffic is typically sufficient

Unlike LLMNR/NBT-NS, mDNS is often a necessary evil for devices and services to function correctly. It is used by so many types of devices that fully blocking inbound mDNS traffic is likely not feasible for every single organization in the modern computing environment. Workers often need tools like Chrome, TeamViewer, screencasting software, etc, to do their jobs. While some organizations already use application controls to block most or all these for security reasons and can get away with blocking mDNS fully, for others, it would harm productivity.

A trade-off can be using some combination of Windows Defender Firewall profiles, application whitelisting, and mitigations that would make a response spoofing attack difficult or near impossible to achieve a result. This would include enabling SMB Signing, LDAP Signing, LDAP Channel Binding, a strong password policy, network segmentation, network access controls that would make it difficult to get a rogue device on the network, least privilege, ensuring privileged users (i.e., Domain Admins) have separate accounts and are not logging into workstations with their privilege account, robust monitoring and alerting, among others.

Security controls have to be applied smartly, and while some organizations will function just fine with a "just shut it off" policy, many need to find a trade-off between security and productivity. Understanding what is being used on your network and maintaining a list of approved applications/services is essential. The risks of mDNS response spoofing are real, but how to mitigate them depends on the organization. It is not as cut and dry as other issues and will likely require some investigation and testing of the proper defense-in-depth strategy appropriate to your operating environment.

Even in well-secured internal networks where LLMNR, NBT-NS, and mDNS have been mitigated, a lesser-known but increasingly relevant attack surface remains: DNS spoofing over IPv6.

DNS Spoofing over IPv6

IPv6 is enabled by default in Windows environments and prioritized over IPv4 but is typically not used or configured. Whenever a Windows computer starts up or is connected to the network, it broadcasts a DHCPv6 request asking for an IPv6 address. Like LLMNR, DHCPv6 is a multicast protocol, so an attacker positioned on the same network segment can respond to these DHCPv6 queries via an attack called DHCPv6 spoofing. The attacker's host provides clients with a specific IP configuration, which includes the address of a rogue DNS server. Once this is done, the attacker can then perform DNS spoofing by forcing each client to query the attacker's rogue DNS server for every DNS resolution. The attacker host then redirects these clients to other rogue servers (such as ntlmrelayx) that are then used to either capture NTLMv1/NTLMv2 password hashes or relay these authentication attempts to SMB, LDAP, etc.

It is not uncommon to encounter an environment that is hardened against LLMNR, NBT-NS, and mDNS response spoofing attacks, but a penetration tester is still able to obtain password hashes or perform NTLM relaying through a DNS spoofing attack over IPv6. Again, this is possible because Windows systems typically prefer IPv6 DNS, which is enabled by default.

Remediating DNS Spoofing over IPv6

IPv6 can be disabled by modifying the following Registry key: HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip6\Parameters\ and setting DisabledComponents to the value 0xFF (or 255 in decimal). This is Microsoft's officially supported method for disabling IPv6 components on all interfaces (both LAN and tunnel adapters). Let's dig into it. We can use a bit of PowerShell to check whether this is partially, fully, or not remediated.

Disabling IPv6 should be well-thought-out, tested, and planned for, as it could break certain things in the network that rely on it. If it is not possible to disable it fully, then other mitigating controls, such as setting the MachineAccountQuota to 0 and enabling both SMB Signing and LDAP signing (which we will discuss in the next section), will greatly lower the risk of leaving it enabled.

Moving On

This section explored various initial access vectors related to response spoofing attacks. These attacks can be used to gather password hashes for offline password cracking, relaying to the SMB service to gain local administrator access to one or more hosts, relaying to the LDAP service to create a computer account, perform enumeration of the domain, or perform an attack resulting in domain compromise.

In the next section, we will look at ways to further harden against NTLM relaying in addition to preventing response spoofing attacks. These settings are important because, even if response spoofing is not possible, other attacks, such as authentication coercion, can be leveraged to relay a valid authentication attempt to hosts and services in the domain.

Initial Access Weaknesses - Post-Spoofing Weaknesses

A successful response spoofing attack can lead to the gathering of NTLMv2 password hashes that can potentially be cracked offline to gain a domain foothold. However, response spoofing can be leveraged in various ways when combined with NTLM relaying, so it is essential to harden the environment further, especially if certain protocols cannot be disabled for business reasons.

NTLM Relaying

NTLM Relaying is a man-in-the-middle (MITM) attack that leverages response spoofing attacks such as LLMNR, NBT-NS, mDNS, and DNS spoofing over IPv6. In this attack, an attacker captures legitimate NTLM authentication traffic from a client and forwards (relays) it to another service such as SMB, LDAP, MSSQL, etc., and then uses it for various other attacks that may lead to individual host or full domain compromise. This type of attack is possible because NTLM does not bind to a specific endpoint, which makes it susceptible to redirection. Some examples where an attacker may be able to leverage NTLM relaying for a foothold include, but are not limited to:

  • Relaying an account that is a local administrator on a given host where SMB signing is disabled to gain local administrator access. If this is a domain-joined host, they will have a low privileged foothold in the domain.
  • Relaying a non-admin user to the SMB service on a target and leveraging this connection to enumerate the domain, possibly obtaining information that can be leveraged for further access
  • Relaying to the LDAP service on a domain controller to create a computer account if the MachineAccountQuota is not set to 0.
  • Relaying to the LDAP service to dump domain information such as users, groups, computers, the password policy, etc., and either finding a password in a description field or then leveraging the user list and password policy for a successful password spray.
  • Relaying to the LDAP service to obtain an interactive LDAP session to perform domain enumeration.
  • Relaying a privileged user account to the LDAP service on a domain controller to perform various other advanced attacks such as Resource-Based Constrained Delegation (RBCD), Shadow Credentials, etc.
  • Relaying to the Web Enrollment endpoint on an Active Directory Certificate Services (AD CS) Certificate Authority server to obtain a certificate for a standard user or even resulting in direct domain compromise.
  • Relaying to an MSSQL server, which then can possibly be leveraged to run operating system commands.

These are some examples, and while more exist, they highlight some of the major risks of NTLM relaying in an AD environment. Mitigating issues such as LLMNR/NBT-NS/mDNS response spoofing, DNS spoofing over IPv6, changing the default MachineAccountQuota, etc., are important steps towards mitigating the risks of NTLM relaying, but further steps can be taken to harden environments against these attacks.

Relaying attacks often require a MITM position in the network, but can still be possible with a forced re-authentication on a target machine, also known as authentication coercion. While outside the scope of this module, coercion attacks are prevalent. Some examples include the PrinterBug, PetitPotam, gaining access to a host, impersonating a logged-in user and forcing authentication on their behalf, leveraging stored procedures in MSSQL after a successful SQL Injection attack, and others. So, even if response spoofing attacks are not possible, an attacker has other options for one of the example relaying attacks above if they gain a foothold through another means.

Remediating NTLM Relaying Attacks

There is no single setting to "disable" NTLM relaying; rather, there is a series of mitigating controls that can be implemented. We will cover some of these later in this section, while others will be covered in future modules.

  • Enforce SMB Signing
  • Disable support for NTLMv1 in favor of NTLMv2
  • Disable NTLM authentication wherever possible in favor of Kerberos. Note that this is typically a massive undertaking and can take considerable time to implement, if even possible, within a given network
  • Enforce LDAP Signing
  • Enforce LDAP Channel Binding
  • Place privileged users in the Protected Users group
  • Disallow NTLM authentication requests from untrusted networks within Active Directory
  • Network segmentation

Default MachineAccountQuota

The MS-DS-Machine-Account-Quota attribute in LDAP controls the number of computer accounts that a user is permitted to create in an Active Directory domain. By default, the value is set to 10, meaning any domain user (regardless of permission level) can join up to 10 computers to the domain. This setting was initially introduced for convenience and scalability, and it helped to reduce the burden on IT departments, especially when AD was first adopted in large organizations. It empowered users to set up their own machines and join them to the domain without having to wait for IT. This came as a benefit for users such as developers, lab admins, and other power users but came with a security tradeoff. When a computer is joined to an Active Directory domain, it is assigned a machine account with a name ending in a dollar sign ($), such as WS01$. This account is associated with a randomly generated password that is securely stored on both the computer (in the Local Security Authority) and within the domain controller's Active Directory database. By default, the computer automatically rotates this password every 30 days, although this interval can be modified through group policy or registry settings.

Offensive tooling exists to allow an attacker to create a computer account that they control by specifying the username and password at the time of creation. This account will exist in LDAP but will not be tied to a physical computer or virtual machine. Computer accounts have most of the rights in Active Directory as a standard domain user account. They can be used to perform credentialed enumeration (i.e., run BloodHound), and various attacks such as Kerberoasting (and other Kerberos attacks), enumerating and attacking Active Directory Certificate Services (AD CS), and many more. An attacker requires valid domain credentials to create a computer account. However, this can be achieved using a response spoofing attack combined with an NTLM relay to the LDAP service on a domain controller. If successful, an attacker will have a valid account to authenticate to Active Directory without having to crack a password hash obtained via a response spoofing attack. This account creation can be seen as legitimate user behavior in environments that are not heavily monitored, allowing an attacker to blend in while performing authenticated enumeration and attacks.

Remediating Default MachineAccountQuota

Remediation is not complex and is critical in hardening Active Directory environments. This will prevent low-privileged users from joining machines to the domain or creating malicious computer accounts. Once this attribute is modified, only delegated users/admins can perform this action. Changing this setting is an important step, but other steps need to be taken, as discussed earlier in this section, to ensure that response spoofing attacks are not possible, which could allow an attacker to relay a privileged user account to create a machine account.

We can check the value of the MachineAccountQuota attribute and set it to 0 using PowerShell one-liners on a domain controller. First, we'll check the setting.

PS C:\htb>Get-ADObject -Identity ((Get-ADDomain).distinguishedname) -Properties ms-DS-MachineAccountQuota

PS C:\htb> Set-ADdomain -Identity wyrmwood.local -Replace @{"ms-DS-MachineAccountQuota"="0"} -Verbose

SMB Signing Disabled

Server Message Block Signing (SMB Signing) is a security feature that verifies both the origin and authenticity of SMB packets. When in place, every SMB message will contain a signature generated via the use of a session key. During SMB communications, the client will place a hash of the entire message in the SMB header signature field. It was first introduced in Windows 2000, Windows NT 4.0, and Windows 98, and the algorithms in use have evolved to be stronger over time. By default, it is disabled on most Windows workstation and server versions but is enforced by default on domain controller hosts. However, it has been enforced by default on Windows 11 since version 24H2.

SMB Signing protects communications by verifying the message hash for tampering and authenticity. If the message changes during transmission (i.e., via a relay attack), the hash will not match, and the message will be rejected. The signature field of the SMB header also confirms both the sender and receiver identity. Enabling SMB signing is essential for preventing SMB relay attacks within a network.

Remediating SMB Signing Disabled

There are three configuration settings possible for SMB signing:

  1. SMB Signing enabled but not required - still vulnerable to SMB relaying
  2. SMB Signing disabled - vulnerable to SMB relaying
  3. SMB Signing enabled and required - not vulnerable to SMB relaying

There are various ways to check if SMB Signing is enabled or disabled, one being the Get-SmbClientConfiguration PowerShell cmdlet on a single host.

LDAP Signing/Channel Binding Disabled

Another key protocol that can be targeted by NTLM Relaying attacks is LDAP. A successful relay to the LDAP service on a domain controller can be used to gather domain information such as users, groups, computers, and their properties, group membership, trusts, domain policy, etc. Relaying to LDAP can also be used to create an attacker-controlled computer account if the ms-DS-MachineAccountQuota is not set to 0, which is a very common manner for gaining a foothold in AD because it does not require an attacker to perform offline password cracking against an NTLMv2 password hash. Relaying to LDAP is even more dangerous when a privileged account is relayed, as it may be used for various attacks that can result in domain compromise, such as creating a new user and granting the account replication privileges. This account can then be used to perform a DCSync attack, obtaining the NTLM password hashes for all accounts in the domain.

When LDAP Signing is enabled, the server will reject LDAP binds that do not request integrity verification through signing. As mentioned previously, unsigned LDAP traffic is susceptible to replay attacks, where an attacker can intercept an authentication attempt and obtain a ticket. This ticket can then be reused to impersonate the legitimate user it was issued for. LDAP Signing applies to LDAP over TCP port 389.

LDAP Channel Binding binds the TLS tunnel and LDAP application layer together, creating a unique fingerprint called a Channel Binding Token (CBT). When enabled, an LDAP communication cannot be reused without establishing a new TLS tunnel. This new tunnel would invalidate the CBT with the initial LDAP communication, essentially preventing relayed authentication. LDAP Channel Binding applies to LDAP over SSL/TLS (also known as LDAPS) on port 636.

It is common for organizations to enable LDAP Signing to prevent relaying to LDAP but overlook LDAP Channel Binding, leaving relays open to LDAPS over port 636. If LDAPS is not enabled, then channel binding isn't needed. However, Microsoft recommends enabling both because LDAPS may be enabled without your knowledge (i.e., by apps or misconfigurations) and for future-proofing or secure defaults in the future.

The table below explains the differences:

Feature Purpose Affected Protocol Required to Stop LDAP Relays?
LDAP Signing Integrity check (mitigates tampering, some relays) LDAP (389) Yes
LDAP Channel Binding Ties TLS to authentication (mitigates MITM/relays over LDAPS) LDAPS (636) Yes, if LDAPS is enabled
Using both Fully mitigate all LDAP relay paths LDAP + LDAPS Yes

Other Common Initial Access Weaknesses

Password Policy Improvements/Weak Passwords - Preventing Password Spraying

The domain password policy is a significant issue still relevant even within medium to large organizations. We see organizations with password policies that are very inviting to a password spraying attack, primarily if an attacker can obtain the password policy through a method such as an SMB null session. We'll cover the major aspects of an AD password policy that may leave an organization open to a password spraying attack. This attack consists of "spraying" a weak/common password against a list of all Active Directory users. If the password policy is known, this can be performed within the account lockout threshold to prevent AD accounts from being locked out. If the password policy is unknown, an attacker may perform a single password spray every few hours or once daily to avoid a potential account lockout (though lockout can still occur).

The key aspects of a domain password policy include the following elements:

  • Minimum Password length - determines how many characters a password must contain.
  • Minimum password age - how long a password must be used before it can be changed.
  • Enforce Password history - the threshold of how many unique passwords must be used before a previous password can be reused. Helps to discourage password reuse.
  • Maximum password age - How many days a password can be used before it must be changed.
  • Password must meet complexity requirements - when enabled, this sets the following requirements: passwords may not contain the user's samAccountName or displayName values, the password must contain characters from three of the four major categories (uppercase letters, lowercase letters, numbers 0-9, non-alphanumeric special characters),
  • Account lockout threshold - determines the number of failed sign-in attempts that will result in user account lockout.
  • Account lockout duration - the number of minutes an account is locked out until automatically becoming unlocked.
  • Reset account lockout counter after - the number of minutes that must pass from a failed login attempt before the lockout counter is reset to 0.
  • Store passwords using reversible encryption - passwords changed or set after this setting is enabled are stored in a retrievable format, not using a one-way hashing function. When an attacker obtains the NTDS file after domain compromise, any passwords set after this setting is enabled can be retrieved in clear text without requiring offline password cracking against the NTLM password hash. One reason for enabling this setting would be to support applications that use protocols requiring the user's clear text password for authentication. It is highly recommended that this setting not be enabled unless absolutely necessary for business reasons, and the environment is extremely well-hardened against attack.

Other options can be modified, but these are the key ones we will focus on.

The following table includes examples of each of these elements, showing the Active Directory defaults, an example of a weak AD password policy, an example of a password policy implemented with Microsoft best practice recommendations, and an example of implementing password policy updates recommended by the National Institute of Standards and Technology (NIST).

Enforce password history 24 passwords 24 passwords No specific requirement (discourages reuse via user training/controls)

Microsoft maintains additional guidelines for password policies in Microsoft 365, which are worth reviewing but are outside the scope of this module, which focuses on on-prem Active Directory. However, elements of these guidelines should be taken into consideration in hybrid environments that integrate on-premises Active Directory domains with Microsoft Entra ID.

It is also possible to specify multiple different password policies in Active Directory through the use of Fine Grained Password Policies. This can be used to define different password lengths, account lockout policies, etc., with stricter settings applies to privileged accounts than standard user accounts. When considering a password policy it is essential to balance security and user experience. You likely would not want standard users to have an account lockout threshold of 1 or 2 attempts, an indefinite lockout period, or a reset lockout counter that does not reset automatically. Some environments may need more stringent requirements, but if these settings are applied across the board too strictly, it could interrupt daily operations and result in a heavy load on the help desk to unlock/reset accounts when users inevitably type them in wrong while going about their day. This also applies to password length and complexity. Longer passwords are not always better; for example, if the password policy requires extremely long passwords, they can be more difficult for users to remember, resulting in them choosing a long but weak password such as Welcome12345 or the like. While long, this type of password may be able to be guessed in a password spraying attack. A better option could be a passphrase or even a random 8 character password that is easy for the user to remember but would be highly unlikely to be guessed in a password spraying attack. Password complexity on its own is not all it's cracked up to be, as it can lead to users choosing predictable passwords that seem strong by substituting characters (i.e., Passw0rd) or having them iterate through sequential passwords whenever a change is required (Password, Password2, Password3, etc.).

However, a fine-grained password policy could be used to enforce much longer passwords, require manual intervention in the case of a lockout, etc., for service accounts and other privileged user accounts in the domain.

Another option that can be implemented to enhance password security is using an enterprise password manager, which will allow users to choose strong, randomized passwords without having to write them down on a sticky note to remember them. Another option is the use of a password filter. If an organization is using AD with Entra ID, then Microsoft Entra Password Protection can be enabled to detect and block weak passwords and their variants. Active Directory supports the implementation of a password filter by using a Password Filter DLL, but this requires secure programming knowledge to ensure that implementing a custom DLL does not introduce additional vulnerabilities, such as sending plaintext passwords across the network. Various commercial Active Directory password filters exist as well. Some, for example, tie into the haveibeenpwned database to prevent users from selecting passwords in public breach databases. These types of password filters also allow administrators to ban common words and variations on those words, no use of first or last name, company name or variations of it, company city, context-specific words (such as the name of a service), repetitive or sequential characters, etc. Implementing the proper password filter solution can significantly improve an organization's overall password strength.

Other Considerations

Some other steps that can be taken include enforcing multi-factor authentication (MFA) wherever possible, not requiring mandatory periodic password resets as this may result in users choosing weak passwords or writing them down, implementing single sign-on (SSO) to allow users to access multiple applications or systems with one set of login credentials, or even transitioning to passwordless authentication through the use of Windows Hello facial recognition or fingerprint scanning, the Microsoft Authenticator app, and physical hardware security keys (i.e., FIDO2 security keys). Various other measures can be taken, and it is worth researching to find the best fit/balance of security and user experience for your environment.

Another critical step is establishing strong monitoring/alerting to quickly detect and stop password spraying attacks. Furthermore, other technical controls/solutions exist that can be used to determine whether an account can authenticate from certain computers and automatically restrict access if an anomalous logon is detected.

On top of solid technical controls is end-user training. People are often the weakest link in an organization, so educating users is critical. One of the most important messages to instill in users is not to reuse their organization password for non-work related services (such as social media or online shopping websites). Furthermore, it is essential that users do not register for external services (outside of those required for their role) with their company email due to the risk of their email and password appearing in a data breach. If this happens, an attacker may be able to leverage a username or email and weak password found in a data breach of a social media site or online service to perform a "credential stuffing" attack and gain access to corporate resources such as email, VPN, or other remote access services. It is common for penetration testers/red teamers to search a company's domain in known data breaches and then attempt to log into external services using these credential pairs, so real-world attackers are doing the same.

Once again, the best password policy for an organization will depend on various factors and differ from company to company. However, an extremely strong and well-thought-out password policy will not matter if layered controls aren't in place on top of hardening against common initial access, lateral movement, and domain privilege escalation vectors. I have personally performed penetration tests where I was able to obtain various NTLMv2 password hashes through a response spoofing attack, but I could not crack them for days with a very powerful GPU password-cracking rig. However, one successful NTLM relay resulted in either a foothold or a direct compromise of the domain. Active Directory is vast and can be challenging to secure, especially in continuously changing and growing environments. Do not rely on just one control, thinking you are safe against attack because you require 14-character passwords while neglecting the basics! It can be tempting to purchase and install tool after tool, but I have pentested plenty of organizations that, once you peel back the proverbial onion, these tools are just bandaids on top of legacy settings and a configuration/patch management nightmare.

Remediating Weak AD Passwords

To enhance password security, administrators can modify the password policy through Group Policy. This involves setting requirements for password complexity, length, and expiration.

This can be done by:

Launching the Group Policy Management Console (GPMC) on a domain controller. Navigate to Computer Configuration -> Policies -> Windows Settings -> Security Settings -> Account Policies -> Password Policy and modify the settings to enforce stronger password requirements following best practices.

Meanwhile, in Computer Configuration -> Policies -> Windows Settings -> Security Settings -> Account Policies -> Account Lockout Policy, administrators can configure appropriate account lockout settings, also per best practices. To sync these settings changes across the domain, a gpupdate /force can be executed.

Now that a hardened password policy is in place across the domain, if weak passwords are identified, administrators can reset them to stronger alternatives. PowerShell provides a secure and efficient method to do this:

PS C:\htb> $NewPassword = Read-Host -AsSecureString "Enter the temporary password for the user"
Set-ADAccountPassword -Identity $accountname -NewPassword $NewPassword -Reset

Administrators can require users to change their passwords at the next login. This ensures that after logging in with the temporary password set by the administrator, users will create their own secure passwords.

PS C:\htb> Set-ADUser -Identity $accountname -ChangePasswordAtLogon $true

Patching Domain Controller Coercion Methods

Multiple modern Active Directory attacks rely on coercion techniques, or tricking domain controllers or other privileged systems into initiating authentication to an attacker-controlled system. These attacks are central to various NTLM relay attacks, allowing an attacker to impersonate a domain controller machine account or other privileged account. Offensive tools exist to take advantage of built-in and legacy protocols such as MS-EFSRPC, MS-RPRN, and various other remote procedure calls. Furthermore, services such as the Print Spooler and the WebClient service (a client-side service used to communicate with WebDav servers) can be leveraged for authentication coercion that can then be relayed to services such as LDAP on a domain controller of the AD CS Web Enrollment endpoint.

One of the most well-known coercion attacks is PetitPotam. This attack allows an attacker to leverage the Encrypting File System Remote Protocol (EFSRPC) to coerce a domain controller to authenticate to a relay server. Microsoft released a patch that should be applied to ensure that this coercion attack cannot be performed as an unauthenticated user because, if not patched, this coercion method can be used for unauthenticated access direct to domain compromise when combined with an unsecured AD CS Web Enrollment endpoint, without other mitigating controls in place. This coercion method (and various others) is still possible with valid domain credentials using built-in/default Remote Procedure Calls (RPCs) when patched. RPC filters can be applied to harden against these coercion methods but are outside the scope of this module.

Aside from ensuring the patch is in place to prevent the PetitPotam coercion attack from an unauthenticated standpoint, the Print Spooler service should be disabled on domain controllers as it leaves them open to coercion attacks using the PrinterBug (or MS-RPRN abuse). On a domain controller we can use PowerShell to check if the Print Spooler service is running.

Vulnerable Services/Applications Leading to Remote Code Execution (RCE)

Another method for gaining a foothold in a domain is by exploiting a vulnerable service or application and either gaining a low privileged session on a domain-joined host and then elevating to NT Authority\SYSTEM (SYSTEM), or the LocalSystem built-in account, or an exploit that results in direct to SYSTEM access if the application or service is running as this all-powerful account. When an attacker obtains SYSTEM-level access on a domain-joined host, this can be leveraged nearly the same as any standard domain user account because this grants them control over a computer account (i.e., WS01$), which can be used for authenticated and enumeration. Below is a real-world scenario to illustrate this example and the importance of patching:

During a penetration test, I was able to exploit a vulnerable version of an application server to gain access as the SYSTEM account. I was then able to leverage this access to obtain the contents of the local SAM database, including the NTLM password hash of the machine account for the domain-joined machine. This was then used to map out the domain using BloodHound and then perform a Kerberoasting attack, resulting in cracking the password for a Domain Admin user due to a weak password policy for privileged users.

Default Credentials or Null Authentication

Weak/default credentials plague even the most hardened environments. When present, this may lead to an attacker being able to abuse built-in functionality to obtain a foothold as a standard domain user (if a service or application is set up to run in their context) or the SYSTEM account per the previous example. Some real-world examples include:

  • Gaining access to a printer or another system that connects to the domain using LDAP. Some systems have an LDAP connection tester that can be configured to point to an attacker-controlled host and then transmit credentials for the LDAP service account in clear text to the attacker.
  • Default credentials or lack of authentication on services such as Tomcat, Jenkins, WebLogic, Splunk, various ManageEngine tools, and many others may result in a foothold via web shell/reverse shell upload or by abusing scripting functionality.

A common scenario during penetration tests is running across an application configured for a demo and then forgotten. I have encountered this many times. Some notable examples include a Splunk Enterprise free trial that converted to a free license when left running. The free version of Splunk does not require authentication. It can be used to deploy a malicious Splunk app, resulting in remote code execution and a foothold if the machine is domain-joined (I have encountered passwordless Splunk running in the context of a Domain Admin!). Another common scenario is finding one of various ManageEngine products with the default admin:admin credential pair. Various of these products can then be used to run operating system commands via scripts or the API on the underlying system itself or systems managed by the tool. Another of these products, used ironically for AD management, can be leveraged to create domain users and even add users to elevated groups.

Every organization should have a documented procedure around demo software, but at the very least, the first step during deployment should be changing the default credentials to avoid leaving a gaping hole in an otherwise well-secured environment. This scenario has played out in countless pentests, where the organization had fixed legacy misconfigurations, disabled all protocols that can be used for response spoofing attacks, and implemented controls that prevented successful password spraying, only to be compromised due to an incredibly simple attack of typing in default credentials.

Open/Writable SMB Shares

Another area to ensure proper hardening is SMB shares. SMB shares are notoriously a pain point for many organizations and can be difficult to audit and gain control over. For this reason, file share permissions are often overlooked and can be a treasure trove for attackers. Open/writable shares can be leveraged in various ways to obtain a foothold. One of the most common techniques involves credential theft through the use of scripts or other malicious files placed on open or writable network shares. In such cases, an attacker can leverage these accessible shares to host files that, when accessed by an unsuspecting user, trigger outbound authentication attempts. This coercion of authentication, often through mechanisms like UNC path references or embedded resource links, can then result in the capture or relay of user credentials, enabling further compromise within the environment.

Below are a few real-world examples to chew on:

  • On a 2-week penetration test, I found the environment to be very well-hardened, and I did not have a foothold by day 9 of the 10-day test. In fact, the only finding I had was an open, writable file share where I had placed a malicious SCF file pointing to my attack host where a Responder instance was running to catch any incoming authentication attempts. Lo and behold, I checked, and a privileged user had visited the share that day, and I obtained their NTLMv2 password hash. I could then crack the password due to a very weak password policy, which kicked off an attack chain that ultimately resulted in domain compromise. One misconfigured share resulted in the "keys to the kingdom."
  • On another penetration test, I found various shares allowing anonymous users to read but not write to them. One of these shares contained backups of virtual machine hard disk (VMDK files). I was able to download one of these, obtain the local SAM database from the system, and dump the password hashes for all local user accounts, including the local administrator account. I then found that this local admin account was used across all systems in the same subnet, and ultimately, I was able to obtain clear text Domain Admin credentials from the Registry of one of these systems.

Auditing file share permissions is critical, and permissions should be set with the principle of least-privilege in mind. One great tool that can be used by Blue Team cybersecurity professionals and penetration testers alike is PowerHuntShares. This PowerShell script will enumerate domain SMB share permissions, identify shares with excessive privileges, shares that provide read/write access, potentially high-risk shares, interesting shares, potential sensitive files, and more. It creates an easy-to-read HTML report and CSV files detailing each finding. Personnel can then review this report against business requirements and make necessary adjustments to harden SMB shares in the environment.

I highly recommend running a tool such as PowerHuntShares (paid, enterprise, tools also exist for these purposes) quarterly to stay on top of potential SMB share issues. Even if no shares exist that can be read/written to without requiring authentication, excessive access may be found for all members of the Domain Users, and similar issues can be shored up to tighten SMB share security.

Remediating Open/Writable SMB Shares

To effectively remediate open or writable SMB shares, it is essential to conduct regular audits (PowerHuntShares can be used) and adjust permissions to adhere to the principle of least privilege. This process involves identifying shares with excessive permissions and modifying them to ensure that only authorized users possess the necessary access. The following outlines the steps for checking and editing SMB share permissions, followed by a concrete example involving Wyrmwood Mining Co.

First, PowerShell can be utilized to list all SMB shares and their associated permissions.

PS C:\htb> # Get all SMB shares
$smbShares = Get-SmbShare

# Iterate over each share and get its permissions
foreach ($share in $smbShares) {
    Write-Host "Share Name: $($share.Name)"
    Write-Host "Path: $($share.Path)"

    # Get the access permissions for the share
    $permissions = Get-SmbShareAccess -Name $share.Name

    # Display the permissions
    foreach ($permission in $permissions) {
        Write-Host "  Account: $($permission.AccountName)"
        Write-Host "  Access: $($permission.AccessControlType)"
        Write-Host "  Rights: $($permission.AccessRight)"
        Write-Host ""
    }

    Write-Host "----------------------------------------"
}

Upon identifying shares with excessive permissions, the Grant-SmbShareAccess cmdlet can be utilized to modify these permissions.

For example, the Revoke-SmbShareAccess cmdlet should be used to remove permissions for unauthorized users or groups.

PS C:\htb> Revoke-SmbShareAccess -Name "ShareName" -AccountName "Domain\UnauthorizedUser"

As for modifying existing permissions, the Set-SmbShareAccess cmdlet should be used.

PS C:\htb>  Grant-SmbShareAccess -Name "ShareName" -AccountName "Domain\User" -AccessRight Read

ASREPRoasting (Do not require Kerberos preauthentication)

When an account in Active Directory is configured with the option Do not require Kerberos preauthentication, an attacker can obtain a Kerberos Ticket Granting Ticket (TGT) for this account. This is not a default setting but is found occasionally in Active Directory environments, typically due to vendor installation guides that specify that a service account must be configured in this way. An attack called ASREPRoasting can be performed as any standard domain user to retrieve an authentication service reply (ASREP) encrypted with the account's passwords. This encrypted TGT can then be subjected to an offline password cracking attack to attempt to retrieve the clear text password if the account has a weak password set.

This attack can also be performed from an unauthenticated standpoint to obtain a foothold in Active Directory. All that is needed is a list of valid domain user accounts, which could be retrieved via one of the unauthenticated recon/enumeration weaknesses discussed earlier or via a username brute-forcing attack. A username brute-forcing attack can be performed against a domain controller without the fear of unlocking accounts. If the domain is configured to use common naming conventions such as jsmith (flast) or john.smith (first.last), then username enumeration can lead to gathering many valid usernames. This can be combined with enumeration of public information such as email addresses and creating username mashups from the company's LinkedIn page to increase the size of the valid user list. If accounts use a non-standard naming convention, for example, a500b or some other randomized string, then username enumeration utilizing this method would be much more difficult unless an attacker can determine the naming convention and write a script to create all possible username combinations. If this is the case, an attacker could obtain a list of almost every account in the domain fairly easily, so there is a tradeoff as it is more difficult to determine a non-standard username naming convention if the environment is extremely well-hardened.

This attack can be performed with various Windows and Linux-based open-source tools and, while less common than other attacks, should be checked for and hardened against unless an account with this setting is essential for business operations. If necessary, the account password should be highly complex (i.e., 24+ characters) to make offline brute-forcing near impossible.

This setting can be easily disabled by unticking the box in Active Directory computers and users. It can also be remediated for either a single user account or all accounts with this setting applied using PowerShell. Below, we are disabling this option for a single account.

PS C:\htb> Get-ADUser -Identity svc-backup | Set-ADAccountControl -DoesNotRequirePreAuth $false -Verbose

PS C:\htb> Get-ADUser -Filter { DoesNotRequirePreAuth -eq $true } | Set-ADAccountControl -DoesNotRequirePreAuth $false -Verbose

Once again, if this setting is required for day-to-day business operations, then it would be best to put in place layered controls to prevent unauthenticated username enumeration, detection alerting/response capabilities to catch an attacker performing username brute forcing, and setting complex passwords for any account with this setting applied. It is also a good idea to periodically run the command Get-ADUSer -Filter { DoesNotRequirePreAuth -eq $true } and determine if any accounts have preauthentication set as not required that no longer need this setting.
